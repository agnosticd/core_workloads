= ocp4_workload_gitlab

== Role Overview

This role deploys GitLab Community Edition on OpenShift with the following features:

* GitLab CE deployment with PostgreSQL and Redis
* Configurable authentication modes (native or Keycloak)
* Automated user creation (both modes)
* Group and repository import with access control
* Keycloak OIDC integration with user pre-creation
* GitLab Runner with Kubernetes executor (optional)
* Automatic SCC (Security Context Constraints) configuration with anyuid
* Configurable persistent storage

== Requirements

* OpenShift 4.x cluster
* Cluster admin access
* For Keycloak mode: Keycloak operator deployed and configured

==== Access Levels

GitLab uses numeric access levels for group membership:

* **guest (10)**: Can view but cannot push
* **reporter (20)**: Can view and create issues
* **developer (30)**: Can push to non-protected branches
* **maintainer (40)**: Can push to protected branches, manage group
* **owner (50)**: Full admin access to the group

==== Branch Protection

The `default_branch_protection` setting controls who can push to default branches (main/master):

* **0**: No protection - Developers can push
* **1**: Partial protection - Developers can push with approval
* **2**: Full protection (default) - Only Maintainers/Owners can push
* **3**: Full protection with MR requirement

Default setting in this role is `0` for workshop environments.

== Usage Examples

=== Example 1: Basic GitLab with Native Authentication

[source,yaml]
----
- name: Deploy GitLab with native auth
  hosts: localhost
  tasks:
  - name: Include GitLab role
    include_role:
      name: ocp4_workload_gitlab
    vars:
      # Root password auto-generated by default
      # To use a custom password (must be 8+ chars):
      # ocp4_workload_gitlab_root_password: "MySecurePassword123"

      ocp4_workload_gitlab_user_count: 5
      ocp4_workload_gitlab_user_base: developer
----

          url: "https://github.com/example/sample-app.git"
          visibility: "private"

      ocp4_workload_gitlab_user_default_access_levels:
      - developer

== Architecture

=== Components

The role deploys the following components:

1. **PostgreSQL** - Database for GitLab
2. **Redis** - Cache and session store
3. **GitLab CE** - Main GitLab application
4. **GitLab Runner** (optional) - CI/CD runner with Kubernetes executor

=== Secrets Created

The role automatically creates the following secrets in the GitLab namespace:

[cols="1,2,3"]
|===
|Secret Name |Keys |Purpose

|`gitlab-secrets`
|`GITLAB_ROOT_PASSWORD`, `POSTGRES_*`, `REDIS_PASSWORD`
|GitLab root password and database credentials

|`gitlab-root-token`
|`token`
|GitLab Personal Access Token (PAT) for automation. This is a long-lived token (365 days) with full API access, used for user and group management during deployment and available for other automation tasks.
|===

NOTE: The `gitlab-root-token` secret contains a Personal Access Token with scopes: `api`, `read_api`, `read_user`, `read_repository`, `write_repository`, `sudo`. This token can be used by external automation tools to interact with the GitLab API.

**Example: Using the token in automation**

[source,bash]
----
# Get the token from the secret
GITLAB_TOKEN=$(oc get secret gitlab-root-token -n gitlab -o jsonpath='{.data.token}' | base64 -d)

# Get GitLab URL
GITLAB_URL=$(oc get route gitlab -n gitlab -o jsonpath='{.spec.host}')

# Use the token to interact with GitLab API
curl -k -H "PRIVATE-TOKEN: ${GITLAB_TOKEN}" https://${GITLAB_URL}/api/v4/projects
----

=== Authentication Modes

==== Native Mode

In native mode, the role:

* Creates local users with username/password
* Creates groups and imports repositories
* Adds users to groups based on `ocp4_workload_gitlab_user_default_access_levels`

==== Keycloak Mode

In Keycloak mode, the role:

* Configures GitLab as an OIDC client in Keycloak
* Pre-creates all users from Keycloak realm with OIDC provider linkage
* Creates groups and imports repositories
* Adds users to groups based on `ocp4_workload_gitlab_user_default_access_levels`
* Users authenticate via Keycloak SSO

=== User and Group Management

**User Creation:**

* **Native mode**: Creates users specified by `user_count` with passwords
* **Keycloak mode**: Pre-creates all users from Keycloak realm (excluding admin, root, ghost) with OIDC linkage

**Group Assignment:**

Users are added to GitLab groups based on the `access_level` defined in `ocp4_workload_gitlab_user_repositories`. The `ocp4_workload_gitlab_user_default_access_levels` variable determines which groups users join automatically.

For example:
```yaml
ocp4_workload_gitlab_user_default_access_levels:
- developer
- maintainer
```

With this configuration, users will be added to all groups where `access_level` is "developer" or "maintainer".

== Troubleshooting

=== Common Issues

==== GitLab pod fails to start with permission errors

**Solution**: Verify that the SCC is properly configured. The role automatically configures `anyuid` SCC which is required for GitLab.

[source,bash]
----
oc get rolebinding gitlab-ce-anyuid -n gitlab -o yaml
oc get sa gitlab-ce -n gitlab -o yaml
----

==== GitLab takes a long time to start

**Solution**: This is normal. GitLab CE can take 5-10 minutes to fully initialize. Monitor the pod logs:

[source,bash]
----
oc logs -f deployment/gitlab -n gitlab
----

==== Repository import fails

**Solution**: Check that the repository URL is accessible from the cluster and verify import status via the GitLab API:

[source,bash]
----
# Get GitLab URL and token
GITLAB_URL=$(oc get route gitlab -n gitlab -o jsonpath='{.spec.host}')
GITLAB_TOKEN=$(oc get secret gitlab-root-token -n gitlab -o jsonpath='{.data.token}' | base64 -d)

# Check import status for all projects
curl -k -H "PRIVATE-TOKEN: ${GITLAB_TOKEN}" "https://${GITLAB_URL}/api/v4/projects" | jq '.[] | {name: .name, import_status: .import_status}'
----

==== Users cannot push to default branch

**Solution**: By default, this role sets `default_branch_protection: 0` to allow developers to push. If you need stricter controls, set `default_branch_protection: 2` (only maintainers) or adjust user access levels to "maintainer" instead of "developer".

[source,yaml]
----
ocp4_workload_gitlab_application_settings:
  default_branch_protection: 0  # 0=no protection, 2=maintainers only
----

=== Debugging

Enable verbose output by setting:

[source,yaml]
----
silent: false
----

Check GitLab health:

[source,bash]
----
# Get GitLab route
GITLAB_URL=$(oc get route gitlab -n gitlab -o jsonpath='{.spec.host}')

# Check liveness
curl -k https://$GITLAB_URL/-/liveness

# Check readiness
curl -k https://$GITLAB_URL/-/readiness
----
