= ocp4_workload_gitlab

== Role Overview

This role deploys GitLab Community Edition on OpenShift with the following features:

* GitLab CE deployment with PostgreSQL and Redis
* Configurable authentication modes (native or Keycloak)
* Automated user creation (native mode)
* Repository import for root and users
* Keycloak group synchronization (Keycloak mode)
* Automatic SCC (Security Context Constraints) configuration with anyuid
* Configurable persistent storage

== Requirements

* OpenShift 4.x cluster
* Cluster admin access
* For Keycloak mode: Keycloak operator deployed and configured

== Architecture

=== Components

The role deploys the following components:

1. **PostgreSQL** - Database for GitLab
2. **Redis** - Cache and session store
3. **GitLab CE** - Main GitLab application
4. **CronJob** (Keycloak mode) - Syncs groups from Keycloak to GitLab

=== Secrets Created

The role automatically creates the following secrets in the GitLab namespace:

[cols="1,2,3"]
|===
|Secret Name |Keys |Purpose

|`gitlab-secrets`
|`GITLAB_ROOT_PASSWORD`, `POSTGRES_*`, `REDIS_PASSWORD`
|GitLab root password and database credentials

|`gitlab-root-token`
|`token`
|GitLab Personal Access Token (PAT) for automation. This is a long-lived token (365 days) with full API access, used by the Keycloak sync cronjob and available for other automation tasks.
|===

NOTE: The `gitlab-root-token` secret contains a Personal Access Token with scopes: `api`, `read_api`, `read_user`, `read_repository`, `write_repository`, `sudo`. This token can be used by external automation tools to interact with the GitLab API.

**Example: Using the token in automation**

[source,bash]
----
# Get the token from the secret
GITLAB_TOKEN=$(oc get secret gitlab-root-token -n gitlab -o jsonpath='{.data.token}' | base64 -d)

# Get GitLab URL
GITLAB_URL=$(oc get route gitlab -n gitlab -o jsonpath='{.spec.host}')

# Use the token to interact with GitLab API
curl -k -H "PRIVATE-TOKEN: ${GITLAB_TOKEN}" https://${GITLAB_URL}/api/v4/projects
----

=== Authentication Modes

==== Native Mode

In native mode, the role:

* Creates local users with username/password
* Assigns users the "developer" access level by default
* Optionally imports repositories for each user

==== Keycloak Mode

In Keycloak mode, the role:

* Configures GitLab as an OIDC client in Keycloak
* Creates groups in Keycloak to match GitLab groups
* Runs a cronjob every minute to sync users from Keycloak groups to GitLab groups
* Assigns appropriate access levels based on group membership

=== Group Sync Process (Keycloak Mode)

The group sync cronjob:

1. Connects to Keycloak admin API
2. For each configured group:
   - Creates the group in Keycloak if it doesn't exist
   - Creates the group in GitLab if it doesn't exist
   - Gets all members of the Keycloak group
   - Adds/updates members in the corresponding GitLab group with the configured access level

== Troubleshooting

=== Common Issues

==== GitLab pod fails to start with permission errors

**Solution**: Verify that the SCC is properly configured. The role automatically configures `anyuid` SCC which is required for GitLab.

[source,bash]
----
oc get rolebinding gitlab-ce-anyuid -n gitlab -o yaml
oc get sa gitlab-ce -n gitlab -o yaml
----

==== GitLab takes a long time to start

**Solution**: This is normal. GitLab CE can take 5-10 minutes to fully initialize. Monitor the pod logs:

[source,bash]
----
oc logs -f deployment/gitlab -n gitlab
----

==== Repository import fails

**Solution**: Check that the repository URL is accessible from the cluster and that the import status:

[source,bash]
----
# Get GitLab pod
POD=$(oc get pod -n gitlab -l app=gitlab -o name)

# Check import status
oc exec -n gitlab $POD -- gitlab-rails runner "Project.all.each { |p| puts \"#{p.name}: #{p.import_status}\" }"
----

==== Keycloak group sync not working

**Solution**: Check the cronjob logs:

[source,bash]
----
# Get latest sync job
JOB=$(oc get jobs -n gitlab -l job-name=gitlab-keycloak-sync -o name | tail -1)

# Check logs
oc logs -n gitlab $JOB
----

=== Debugging

Enable verbose output by setting:

[source,yaml]
----
silent: false
----

Check GitLab health:

[source,bash]
----
# Get GitLab route
GITLAB_URL=$(oc get route gitlab -n gitlab -o jsonpath='{.spec.host}')

# Check liveness
curl -k https://$GITLAB_URL/-/liveness

# Check readiness
curl -k https://$GITLAB_URL/-/readiness
----
