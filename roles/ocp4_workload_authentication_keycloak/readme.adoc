= ocp4_workload_authentication_keycloak

== Role Overview

This role deploys Red Hat build of Keycloak (RHBK) operator and configures it for OpenShift authentication with the following features:

* Keycloak operator deployment
* PostgreSQL database for Keycloak
* Realm creation with OpenShift OAuth integration
* User and admin account creation
* Support for additional OAuth clients (e.g., GitLab, other apps)
* Configurable groups for user organization

== Requirements

* OpenShift 4.x cluster
* Cluster admin access

=== Additional Clients Configuration

The role supports creating additional OAuth clients for applications like GitLab, Grafana, etc.

[cols="1,1,1,3"]
|===
|Variable |Default |Type |Description

|ocp4_workload_authentication_keycloak_clients
|[]
|List
|Additional OAuth clients to create in the realm
|===

Each client entry should have:

* `clientId`: Client ID (required)
* `secret`: Client secret (required)
* `redirectUris`: List of redirect URIs (optional, defaults to `["*"]`)
* `description`: Client description (optional)
* `publicClient`: true/false (optional, defaults to false)
* `standardFlowEnabled`: true/false (optional, defaults to true)
* `directAccessGrantsEnabled`: true/false (optional, defaults to true)
* `serviceAccountsEnabled`: true/false (optional, defaults to true)

**Example:**

[source,yaml]
----
ocp4_workload_authentication_keycloak_clients:
- clientId: gitlab
  secret: "{{ lookup('password', '/dev/null chars=ascii_letters,digits length=32') }}"
  redirectUris:
  - "https://gitlab-{{ guid }}.{{ openshift_cluster_ingress_domain }}/*"
  description: "GitLab OAuth Client"
- clientId: grafana
  secret: "grafana-secret-123"
  redirectUris:
  - "https://grafana.apps.example.com/login/generic_oauth"
  description: "Grafana OAuth Client"
----

=== Groups Configuration

The role creates default `admins` and `users` groups, plus any additional groups specified.

[cols="1,1,1,3"]
|===
|Variable |Default |Type |Description

|ocp4_workload_authentication_keycloak_admin_groups
|[]
|List
|Groups to create and assign to admin user

|ocp4_workload_authentication_keycloak_user_groups
|[]
|List
|Groups to create and assign to all regular users
|===

Each group entry should have:

* `name`: Group name (required)
* `path`: Group path (optional, defaults to /name)
* `realmRoles`: List of realm roles to assign (optional)
* `attributes`: Dict of group attributes (optional)

**Example:**

[source,yaml]
----
# Groups for admin user
ocp4_workload_authentication_keycloak_admin_groups:
- name: cluster-admins
  realmRoles:
  - admin
- name: operations
  realmRoles:
  - admin
  attributes:
    team: ["platform"]

# Groups for regular users
ocp4_workload_authentication_keycloak_user_groups:
- name: developers
  realmRoles:
  - user
- name: team-leads
  realmRoles:
  - user
  attributes:
    department: ["engineering"]
----

== Usage Examples

=== Example 1: Basic Keycloak with Default Configuration

[source,yaml]
----
- name: Deploy Keycloak
  hosts: localhost
  tasks:
  - name: Include Keycloak role
    include_role:
      name: ocp4_workload_authentication_keycloak
    vars:
      ocp4_workload_authentication_keycloak_num_users: 5
----

=== Example 2: Keycloak with GitLab Client

[source,yaml]
----
- name: Deploy Keycloak with GitLab client
  hosts: localhost
  tasks:
  - name: Include Keycloak role
    include_role:
      name: ocp4_workload_authentication_keycloak
    vars:
      ocp4_workload_authentication_keycloak_num_users: 10
      ocp4_workload_authentication_keycloak_user_username_base: student

      # Create GitLab OAuth client
      ocp4_workload_authentication_keycloak_clients:
      - clientId: gitlab
        secret: "{{ lookup('password', '/dev/null chars=ascii_letters,digits length=32') }}"
        redirectUris:
        - "https://gitlab-gitlab.{{ openshift_cluster_ingress_domain }}/*"
        description: "GitLab SSO Integration"

      # Create user groups that GitLab will sync
      ocp4_workload_authentication_keycloak_user_groups:
      - name: developers
        realmRoles:
        - user
      - name: maintainers
        realmRoles:
        - user
----

=== Example 3: Keycloak with Separate Admin and User Groups

[source,yaml]
----
- name: Deploy Keycloak with custom groups
  hosts: localhost
  tasks:
  - name: Include Keycloak role
    include_role:
      name: ocp4_workload_authentication_keycloak
    vars:
      ocp4_workload_authentication_keycloak_num_users: 20

      # Admin groups - assigned to admin user only
      ocp4_workload_authentication_keycloak_admin_groups:
      - name: cluster-admins
        realmRoles:
        - admin
      - name: operations
        realmRoles:
        - admin
        attributes:
          team: ["platform"]

      # User groups - assigned to all regular users
      ocp4_workload_authentication_keycloak_user_groups:
      - name: developers
        realmRoles:
        - user
      - name: team-leads
        realmRoles:
        - user
        attributes:
          level: ["senior"]

      # Result:
      # - Regular users will be in: /users, /developers, /team-leads
      # - Admin will be in: /admins, /cluster-admins, /operations
----

== Architecture

The role deploys:

1. **Keycloak Operator** - Manages Keycloak instances
2. **PostgreSQL** - Database backend for Keycloak
3. **Keycloak Instance** - Main Keycloak server
4. **Realm Import** - Configured realm with:
   - OpenShift OAuth client (automatic)
   - Additional OAuth clients (optional)
   - Users and admin accounts
   - Groups (default + custom)
   - Realm roles

== OAuth Client Integration

When integrating applications with Keycloak:

1. **Add the client to the configuration** using `ocp4_workload_authentication_keycloak_clients`
2. **Configure the application** to use Keycloak OAuth
3. **Set redirect URIs** to match your application's callback URLs

**Common OAuth endpoints:**

* Authorization: `https://keycloak-{namespace}.{cluster-domain}/realms/{realm}/protocol/openid-connect/auth`
* Token: `https://keycloak-{namespace}.{cluster-domain}/realms/{realm}/protocol/openid-connect/token`
* UserInfo: `https://keycloak-{namespace}.{cluster-domain}/realms/{realm}/protocol/openid-connect/userinfo`

== Group Management

Groups can be used to:

* Organize users by role or team
* Assign permissions via realm roles
* Store custom attributes for applications
* Control access to specific resources

Applications like GitLab can sync these groups and use them for access control.

=== User to Group Assignment

**Automatic Group Assignment:**

Users are automatically assigned to groups based on their role:

* **Regular users** → `/users` group + all groups in `ocp4_workload_authentication_keycloak_user_groups`
* **Admin user** → `/admins` group + all groups in `ocp4_workload_authentication_keycloak_admin_groups`

**Example:**

[source,yaml]
----
# Admin groups - assigned to admin user
ocp4_workload_authentication_keycloak_admin_groups:
- name: cluster-admins
  realmRoles:
  - admin
- name: operations
  realmRoles:
  - admin

# User groups - assigned to all regular users
ocp4_workload_authentication_keycloak_user_groups:
- name: developers
  realmRoles:
  - user
- name: team-leads
  realmRoles:
  - user

# Result:
# - Regular users will be in: /users, /developers, /team-leads
# - Admin will be in: /admins, /cluster-admins, /operations
----

**Important Notes:**

* Groups defined in the arrays are both **created** and **assigned** to users
* Group paths are automatically prefixed with `/` (e.g., `developers` becomes `/developers`)
* Users inherit realm roles from their groups
* Group membership is set at user creation time in the realm import
* This is ideal for applications like GitLab that sync Keycloak groups for access control

== Troubleshooting

=== Keycloak pod fails to start

**Solution**: Check PostgreSQL is running and accessible.

[source,bash]
----
oc get pods -n keycloak
oc logs -n keycloak deployment/postgresql
----

=== OAuth client not working

**Solution**: Verify redirect URIs match exactly, including trailing slashes.

[source,bash]
----
# Get the realm configuration
oc get keycloakrealmimport sso -n keycloak -o yaml
----

=== Users not appearing in realm

**Solution**: Check the KeycloakRealmImport status.

[source,bash]
----
oc get keycloakrealmimport -n keycloak
oc describe keycloakrealmimport sso -n keycloak
----
